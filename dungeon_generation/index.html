# DungeonScript DSL Specification

## 1. OVERVIEW

DungeonScript is a declarative domain-specific language for procedural dungeon generation on a 32×32 grid. It enforces a **partition-then-populate-then-connect** pipeline that prevents invalid topologies while enabling diverse generation strategies—from structured BSP room layouts to organic cellular automata caves. The DSL exposes continuous parameters (twistiness, density) and range-based sizing to support automated optimization of the playability_score metric (reachability + room_size_stddev + corridor_turns + room_density).

## 2. OPERATORS

| Name | Args | Semantics | Example |
|------|------|-----------|---------|
| `INIT` | `fill` | Initialize 32×32 grid; 0=floor, 1=wall | `INIT(1)` |
| `PARTITION` | `strategy, depth` | Subdivide space; strategy ∈ {bsp, grid, random} | `PARTITION(bsp, 4)` |
| `ROOM` | `w_range, h_range, placement` | Carve room in partition; placement ∈ {partition, random, center} | `ROOM(4-8, 4-8, partition)` |
| `CORRIDOR` | `mode, twistiness` | Connect rooms; mode ∈ {mst, nearest, sequential}; twist 0.0-1.0 | `CORRIDOR(mst, 0.3)` |
| `WALK` | `steps, width` | Drunkard's walk carving from center; width 1-3 | `WALK(200, 2)` |
| `CELLULAR` | `iterations, birth, survive` | Cellular automata smoothing pass | `CELLULAR(3, 5, 4)` |
| `NOISE` | `density` | Scatter floor tiles randomly; density 0.0-1.0 | `NOISE(0.45)` |
| `DOOR` | `style` | Place doors; style ∈ {single, random, none} | `DOOR(single)` |
| `CONSTRAINT` | `key, min, max` | Set bounds: rooms, room_size, corridor_length | `CONSTRAINT(rooms, 5, 12)` |
| `POSTPROCESS` | `op` | Cleanup; op ∈ {connect, smooth, remove_deadends} | `POSTPROCESS(connect)` |
| `REPEAT` | `n, operator` | Execute operator n times with variation | `REPEAT(6, ROOM(3-6, 3-6, random))` |
| `FILL` | `region, value` | Set region to wall/floor; region ∈ {all, border} | `FILL(border, 1)` |

## 3. GRAMMAR

```
program     ::= "generate:" statement+
statement   ::= operator | repeat | constraint
operator    ::= INIT(int) | PARTITION(strategy, int) | ROOM(range, range, placement)
              | CORRIDOR(mode, float) | WALK(int, int) | CELLULAR(int, int, int)
              | NOISE(float) | DOOR(style) | POSTPROCESS(op) | FILL(region, int)
repeat      ::= REPEAT(int, operator)
constraint  ::= CONSTRAINT(key, int, int)
range       ::= int "-" int | int
strategy    ::= bsp | grid | random
placement   ::= partition | random | center
mode        ::= mst | nearest | sequential
style       ::= single | random | none
op          ::= connect | smooth | remove_deadends
```

**Ordering Convention**: INIT → CONSTRAINT → (PARTITION|NOISE|WALK) → ROOM → CORRIDOR → DOOR → POSTPROCESS

## 4. COMPILATION TARGET

The DSL compiles to a Python function with signature:

```python
def generate(width: int = 32, height: int = 32, seed: int | None = None) -> np.ndarray:
    """Returns a 2D numpy array of shape (height, width) with 0=floor, 1=wall."""
```

The compiler translates each operator to imperative grid mutations. `POSTPROCESS(connect)` guarantees all floor tiles are reachable. The function must be deterministic given a seed.

## 5. EXAMPLE PROGRAMS

**BSP Rooms with MST Corridors** (structured, high room_density):
```
generate:
  INIT(1)
  CONSTRAINT(rooms, 6, 12)
  PARTITION(bsp, 4)
  ROOM(5-9, 4-8, partition)
  CORRIDOR(mst, 0.4)
  DOOR(single)
  POSTPROCESS(smooth)
  POSTPROCESS(connect)
```

**Cellular Automata Cave** (organic, variable room_size_stddev):
```
generate:
  INIT(1)
  NOISE(0.45)
  CELLULAR(4, 5, 4)
  REPEAT(3, WALK(100, 2))
  POSTPROCESS(remove_deadends)
  POSTPROCESS(connect)
  CONSTRAINT(rooms, 1, 8)
```
